Um Daten über dein NINA-B4-Modul (das ein u-blox Bluetooth-Modul ist) zu senden, hängt der Befehl davon ab, ob du Daten über ein bestimmtes GATT-Characteristic schreiben möchtest oder ob du den transparenten Datenmodus (SPS - Serial Port Service) nutzt. Da du in deinem Setup bereits mit AT+UBTGW (GATT Write) arbeitest, möchtest du sehr wahrscheinlich eine GATT-Characteristic beschreiben. Hier sind die passenden AT-Befehle dafür: 1. Der Standard-Befehl: GATT Write (AT+UBTGW) Du nutzt bereits AT+UBTGW=0,33,0100 (was meistens das Aktivieren von Benachrichtigungen/CCCD ist). Um eigene Daten an einen Server zu senden, nutzt du genau denselben Befehl, übergibst aber die Nutzdaten als Hex-String. Syntax: Plaintext AT+UBTGW=[conn_handle],[value_handle],[hex_value] conn_handle: Die Verbindungskennung (in deinem Fall 0). value_handle: Das Handle der Characteristic, in die du schreiben willst (z. B. 35). hex_value: Die Daten, die du senden willst, in Hexadezimalform. Beispiel: Wenn du das Wort "Hello" senden möchtest (Hex: 48656C6C6F) an das Handle 35: Plaintext AT+UBTGW=0,35,48656C6C6F 2. Alternative: GATT Write Without Response (AT+UBTGWW) Wenn die Gegenstelle (der Server) keine Bestätigung für den Erhalt der Daten senden muss (was schneller ist und oft für fortlaufende Datenströme genutzt wird), verwendest du AT+UBTGWW. Syntax: Plaintext AT+UBTGWW=[conn_handle],[value_handle],[hex_value] Beispiel: Plaintext AT+UBTGWW=0,35,48656C6C6F 3. Der "Gegenspieler" zu deinem Befehl: GATT Notification (AT+UBTGN) Hinweis: Das brauchst du nur, wenn dein NINA-B4 als GATT-Server agiert und Daten an einen Client (z. B. eine Smartphone-App) pushen soll. Falls du Daten von lokalen Handles an die Gegenstelle senden willst, nutzt du: Plaintext AT+UBTGN=[conn_handle],[value_handle],[hex_value] Zusammenfassung für deinen Ablauf: Finde das korrekte value_handle für die Schreib-Eigenschaft (Write oder Write Without Response) heraus (notfalls mit AT+UBTGDC alle Characteristics abfragen). Wandle deinen Text/deine Daten in Hex um. Sende es mit AT+UBTGW=0,[Handle],[Hexdaten]. AT+UBTGW=0,32,48656C6C6F so klappt es. aber wie komme ich auf die 32? Dass es mit der 32 klappt, ist super! Aber die Frage, wie man auf diese magische Zahl (das sogenannte Attribute Handle) kommt, ist im BLE-Bereich essenziell. Kurz gesagt: Ein BLE-Gerät (Server) ist wie ein Hotel aufgebaut. Das Hotel hat Services (Etagen) und Characteristics (Zimmer). Das Handle ist schlichtweg die genaue Zimmernummer. Da du diese Nummern im Vorfeld oft nicht kennst, musst du das Gerät danach fragen (Discovery). Hier sind die drei Wege, wie du an die 32 (oder andere Handles) kommst: Weg 1: Der "Alles-Anzeiger" (Discovery über AT-Befehle) Wenn du mit dem Modul verbunden bist (AT+UBTACLC=...), kannst du die Gegenstelle bitten, dir ihre komplette Struktur offenzulegen. Nutze dafür den Befehl AT+UBTGDC (GATT Discover Characteristics): Plaintext AT+UBTGDC=0 (Die 0 steht wieder für dein conn_handle) Das Modul antwortet dir dann mit einer Liste, die ungefähr so aussieht: Plaintext +UBTGDC:0,31,32,02,77,65,72,64,65,66,67,68,69,70,71,72,73,74,75,76 Wie liest man diese Antwort? Die Parameter sind wie folgt aufgebaut: +UBTGDC:[conn_handle],[decl_handle],[value_handle],[properties],[UUID] 31 (decl_handle): Das Handle der Eigenschafts-Deklaration. 32 (value_handle): Das ist deine gesuchte Zahl! Das ist das exakte Handle, in das die Daten geschrieben oder aus dem sie gelesen werden. 02 (properties): Ein Bitmask-Wert, der dir sagt, was das Zimmer erlaubt (z.B. Read, Write, Notify). Der Rest (77,65...): Die UUID der Characteristic in Hex (z. B. die standardisierte 128-Bit- oder 16-Bit-UUID). Weg 2: Gezielt nach einer bestimmten UUID suchen Wenn du die UUID der Eigenschaft kennst (z. B. aus dem Datenblatt des Sensors oder deiner eigenen App), musst du nicht die ganze Liste durchwühlen. Du kannst gezielt nach dem Handle für diese UUID suchen. Syntax: Plaintext AT+UBTGDCC=[conn_handle],[uuid] Beispiel: Wenn die UUID deiner Schreib-Characteristic FFF2 ist: Plaintext AT+UBTGDCC=0,FFF2 Das Modul liefert dir dann direkt die Zeile zurück, in der du dein value_handle (die 32) ablesen kannst. Weg 3: Der bequeme Weg (Smartphone-App) Wenn du beim Entwickeln am PC parallel sehen willst, was los ist, kannst du eine App wie nRF Connect (von Nordic Semiconductor) oder LightBlue auf dem Smartphone nutzen: Verbinde dich mit dem Smartphone mit dem Zielgerät. Suche den entsprechenden Service und die Characteristic heraus. In Apps wie nRF Connect steht das Handle oft direkt in Klammern neben der Eigenschaft (oft als Hex angegeben, z. B. (0x0020) – was umgerechnet in Dezimal genau 32 ist). Zusammenfassung für deinen Code: Bevor du also Daten sendest, läuft der saubere BLE-Workflow so ab: Verbinden (AT+UBTACLC) → Characteristics abfragen (AT+UBTGDC) → Handle der passenden UUID herausfiltern → Daten senden (AT+UBTGW). AT+UBTGW=0,33,0100 was macht das? Der Befehl AT+UBTGW=0,33,0100 bedeutet übersetzt: „Ich möchte die automatischen Benachrichtigungen für einen bestimmten Datenstrom einschalten.“ Technisch gesehen machst du damit ein GATT Write auf das sogenannte CCCD (Client Characteristic Configuration Descriptor). Hier ist die genaue Zerlegung, was dort im Hintergrund passiert: Die Parameter im Detail 0 (conn_handle): Die ID deiner bestehenden Bluetooth-Verbindung. 33 (value_handle): Das Handle des Konfigurations-Deskriptors (CCCD). Dieses liegt in der BLE-Struktur fast immer direkt hinter dem eigentlichen Daten-Handle (32). 0100 (hex_value): Das ist das Aktivierungs-Bit-Pair im Little-Endian-Format. 0100 steht für: Notifications (Benachrichtigungen) aktivieren. (Hättest du 0200 gesendet, würde das „Indications“ aktivieren; 0000 schaltet es wieder aus). Warum musst du das machen? In der BLE-Welt darf ein Server (dein NINA-Modul oder der Sensor auf der Gegenseite) nicht einfach ungefragt Daten senden. Das würde zu viel Strom verbrauchen. Der Server wartet, bis der Client (z. B. deine Software) explizit sagt: „Ich bin bereit und höre ab jetzt zu.“ Vor dem Befehl: Die Gegenstelle hat neue Messwerte oder Daten, behält sie aber für sich. Du sendest AT+UBTGW=0,33,0100: Das „Ventil“ wird geöffnet. Nach dem Befehl: Sobald auf Handle 32 neue Daten eingehen, pusht das Modul diese sofort automatisch an dich raus. Du empfangst sie dann als unaufgeforderte Antwort, die meistens so aussieht: +UBTGRN:0,32,48656C6C6F... (GATT Notification erhalten). Es ist also quasi der Startschuss für den Datenempfang. was ist das daten-handle? Das Daten-Handle (oft auch Value Handle genannt) ist im Grunde die eindeutige Adresse einer bestimmten Information innerhalb eines Bluetooth-Geräts. Wenn man ein BLE-Gerät ausliest, gibt es zwei eng miteinander verknüpfte Adressen: das Deklarations-Handle und das Daten-Handle. Hier ist die genaue Erklärung, was das Daten-Handle macht und wie es sich unterscheidet: Das Hotel-Gleichnis Stell dir das Bluetooth-Gerät wie ein Hotel vor: Die UUID ist der Name des Zimmers (z. B. „Suiten-Nummer für Temperaturmessung“). UUIDs sind oft sehr lang (z. B. 128-Bit-Hex-Zahlen) und weltweit eindeutig. Das Daten-Handle ist die Zimmernummer (z. B. Raum 32). Da es extrem ineffizient wäre, wenn der Mikrocontroller bei jedem Senden oder Empfangen von Daten die lange UUID (z. B. 6E400002-B5A3-F393-E0A9-E50E24DCCA9E) mitschicken müsste, handeln die Geräte beim Verbindungsaufbau einmalig die kurzen Daten-Handles aus. Ab dann läuft die Kommunikation nur noch über diese kurze Nummer. Metapher 31 Deklarations-Handle (Declaration) Die Metadaten. Hier steht: „Achtung, das folgende Zimmer hat die UUID XYZ und man darf darin nur schreiben.“ Das Türschild, auf dem steht, wer hier wohnt und was erlaubt ist. 32 Daten-Handle (Value) Die echten Nutzdaten. Wenn du hier hineinschreibst, änderst du den Wert. Wenn du hieraus liest, kriegst du den aktuellen Wert. Der Inhalt des Zimmers (das eigentliche Bett oder die Möbel). Warum ist das wichtig für dich? Wenn du mit deinem NINA-B4-Modul kommunizierst, musst du für Aktionen wie AT+UBTGW (Schreiben) oder AT+UBTGR (Lesen) immer das Daten-Handle ansprechen, niemals das Deklarations-Handle. Die 32 ist also der direkte Datenkanal für dieses spezifische Feature deines Bluetooth-Gegenstücks. ist das immer 32? Nein, das ist keineswegs immer die 32. Das Daten-Handle ist dynamisch und kann sich von Gerät zu Gerät – und manchmal sogar nach einem Neustart oder Firmware-Update desselben Geräts – komplett ändern. Dass es bei dir aktuell die 32 ist, ist reiner Zufall der internen Speicherstruktur des Bluetooth-Geräts, mit dem du dich verbindest. Hier ist der Grund, warum sich diese Nummern ändern: 1. Jedes Gerät baut sein "Hotel" anders auf Wenn du dich statt mit deinem aktuellen Bluetooth-Modul mit einem anderen Sensor verbindest (z. B. einem Herzfrequenzgurt, einem Temperatursensor oder einem anderen NINA-Modul), hat dieser Entwickler vielleicht mehr oder weniger Funktionen (Characteristics) eingebaut. Hat das andere Gerät mehr Funktionen, rutscht dein Daten-Handle vielleicht auf die 45. Hat es weniger Funktionen, ist es vielleicht schon die 12. 2. Die Reihenfolge bestimmt die Nummer Die Handles werden beim Starten des Bluetooth-Gegenstücks einfach von oben nach unten durchnummeriert (meistens beginnend bei 1). Wenn der Entwickler der Gegenseite ein Firmware-Update aufspielt und eine einzige neue Funktion vor deiner aktuellen Funktion einfügt, verschieben sich alle nachfolgenden Handles nach hinten. Aus der 32 wird dann plötzlich die 34. Die goldene Regel für stabile Software Weil die Handles so unzuverlässig sind, darf man sie niemals fest im Programmcode einbetten (kein "Hardcoding"), wenn man eine App oder Firmware schreibt, die mit verschiedenen Geräten funktionieren soll. FFF2 oder ein langer 128-Bit-String). Das machst du mit AT+UBTGDC=0. Handle merken: Dein Mikrocontroller durchsucht die Antwort nach der UUID, greift sich die dazugehörige Daten-Handle-Nummer (aktuell die 32) und speichert sie in einer Variablen. Nutzen: Erst jetzt schickst du dein AT+UBTGW=0,[Variable],48656C6C6F. So ist es deiner Software völlig egal, ob das Daten-Handle heute 32, morgen 45 oder nächste Woche 12 ist. Der Aufbau des Events sieht so aus: +UUBTGRW:[conn_handle],[value_handle],[hex_value] In deinem Fall sind die Nutzdaten das Hex-Paar 00 (was dem Wert 0 entspricht). Hier ist eine robuste C-Funktion, die den String parst, die Hex-Nutzdaten extrahiert und sie sauber als uint32_t zurückgibt. Da BLE-Daten standardmäßig in Little-Endian (niederwertigstes Byte zuerst) übertragen werden, wandelt die Funktion die Hex-Bytes auch direkt in die richtige CPU-Reihenfolge um. "+UUBTGRW:0,32,AABBCCDD,1") * @return uint32_t Die konvertierten Nutzdaten (0 bei Fehler oder leeren Daten) */ uint32_t parse_uubtgrw_payload(const char *event_str) { char hex_buf[32] = {0}; int conn_handle = 0; int value_handle = 0; // Suchen nach dem Start des relevanten Inhalts hinter dem Doppelpunkt const char *data_start = strchr(event_str, ':'); if (!data_start) return 0; data_start++; // Springe hinter den Doppelpunkt // Parsen von conn_handle, value_handle und Extrahieren des Hex-Strings. // %31[^,] liest maximal 31 Zeichen bis zum nächsten Komma ein (die Nutzdaten). int parsed = sscanf(data_start, "%d,%d,%31[^,]", &conn_handle, &value_handle, hex_buf); // Wenn wir nicht mindestens die 3 Elemente extrahieren konnten, stimmt das Format nicht if (parsed < 3) { return 0; } size_t hex_len = strlen(hex_buf); if (hex_len == 0 || hex_len % 2 != 0) { return 0; // Ungültige oder unvollständige Hex-Daten } uint32_t result = 0; // BLE überträgt Daten in Little-Endian (z.B. "01020304" ankommt -> mathematisch 0x04030201) // Wir lesen die Hex-Paare von hinten nach vorne ein, um es in die native CPU-Reihenfolge zu bringen. int shift = 0; for (int i = (int)hex_len - 2; i >= 0; i -= 2) { char byte_str[3] = { hex_buf[i], hex_buf[i+1], '\0' }; uint8_t byte_val = (uint8_t)strtol(byte_str, NULL, 16); result |= ((uint32_t)byte_val << shift); shift += 8; // Schutz vor Buffer-Overflow, falls mehr als 4 Bytes (8 Hex-Zeichen) geschickt wurden if (shift >= 32) break; } return result; } // === Test-Code === int main() { // Test 1: Dein originaler String (1 Byte Nutzdaten: 0x00) const char *test1 = "+UUBTGRW:0,32,00,1"; printf("Ergebnis 1: %u (Hex: 0x%08X)\n", parse_uubtgrw_payload(test1), parse_uubtgrw_payload(test1)); // Test 2: 2 Bytes Nutzdaten (Little Endian: "E803" -> 0x03E8 = 1000) const char *test2 = "+UUBTGRW:0,32,E803,1"; printf("Ergebnis 2: %u (Hex: 0x%08X)\n", parse_uubtgrw_payload(test2), parse_uubtgrw_payload(test2)); // Test 3: 4 Bytes Nutzdaten (Little Endian: "78563412" -> 0x12345678) const char *test3 = "+UUBTGRW:0,32,78563412,1"; printf("Ergebnis 3: %u (Hex: 0x%08X)\n", parse_uubtgrw_payload(test3), parse_uubtgrw_payload(test3)); return 0; } Funktionsweise kurz erklärt: sscanf(..., "%d,%d,%31[^,]"): Überspringt die ersten beiden Zahlen (conn_handle und value_handle) und schneidet die Hex-Nutzdaten genau beim darauffolgenden Komma ab. Hier sind die zwei Anpassungen, die du in deiner index.html vornehmen musst, um das Problem zu lösen: Anpassung 1: Auf "Without Response" umstellen (Sehr wahrscheinlich die Lösung) Ersetze in deiner index.html im sendBtn-Eventlistener die Zeile mit writeValue durch writeValueWithoutResponse. Das erzwingt, dass der Browser die Daten direkt und ohne das Warten auf ein GATT-Acknowledge herausschickt – genau so, wie es schlanke BLE-Profile erwarten.
